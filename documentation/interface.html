System Overview

NPS is developed in Allaire ColdFusion with Microsoft Access 2000 as the database engine. It is a project management system for REAS acquisition, disposal and consulting projects. The primary users are REAS advisors who input and maintain the data for their own projects. The system is also used by management to gain an overview of current and completed projects as well as finding information about specific projects.

In order to ensure that the system is useful for the advisors, NPS has been designed to be simple and intuitive. It was decided that a web application would provide the most familiar interface and would be the most widely accessible. It was also important to ensure that using the system would not be a burden to the advisors, so the amount of data maintained about each project has been kept to a minimum.

To give management an overview of the current and completed projects, the system includes a basic reporting function. The user can search for projects using several different criteria and produce summary reports based on the results.

There is also a separate administration section where users with the appropriate access level can maintain the data that appears in many of the drop-down lists. Admin users can also download data from the database tables in ASCII format.

  • Login/Logout

    In order for a user to add or edit any data they must be logged in. Only users with the "read/write" or "admin" access levels can log in. To log in, the user simply clicks on "Menu" and then "Login". Once at the Login page the user simply has to select their name from a list and enter the password and they will be returned to the Project Search page.

    To log out, the use clicks on "Menu" and then "Logout". The "Logout" option is not available when the user is viewing or editing a project. When the user clicks on "Logout" they will be returned to the Project Search page.

  • Search and Sort

    The Project Search page is the first page the user sees when they access the system. It allows the user to filter projects by project status, type, location, description, advisor, client, project number, file number, and client contact. If the user is logged in, their name will automatically be selected in the "Advisor" drop-down list. Also, only items which actually exist as data for at least one project will be available in the list-boxes (i.e. if there are no projects with Vancouver as the location, Vancouver will not be displayed in the "Location" drop-down list). When they have selected their criteria from the drop-down lists and/or entered their criteria in the text boxes, they click "Submit" to display the results on the Search Results page.

    To sort the results the user can click on the column headings on the Search Results page. If they want to sort by two fields they can click on a second column which becomes the primary sort, and the first column becomes the secondary sort.

  • Project Display and Edit

    Projects fall into one of three categories: acquisitions, disposals and consulting projects. When the user wants to update a project, they perform a simple search (see above) and click on the "Details" button for the project of their choice. Depending on the type of the selected project, this will take them to the appropriate project page (either acquisitions.cfm, disposals.cfm, or consulting.cfm) in which they can flip through the tabs and add to, or edit, the project's data. When they are done they can either click on "Save" or "Cancel" to return to the search results and select another project. If the user is not logged in, only the "Cancel" button will be available and they will not be able to save any changes.

    There are two types of data and, thus, two types of data entry. There is some data that a project only has one of. For example, a project only has one description and one location. This type of data is simply entered into text boxes or selected from drop-down lists. The other type of data is data that a project can have many of. For example, a project can have many status updates and many appraisals. This second type of data requires the user to click on an add or edit button which will take them to another page. On this secondary page, they will enter the data into text boxes and drop-down lists and then click submit to add the new data or edit already existing data.

  • Print

    There are two types of information that can be printed: information concerning a particular project, and information concerning many projects. To print information concerning a particular project, the user must go to that project by performing a search (see above) and clicking on the "Details" button for that project (see above). Once at the project, the user clicks on "Print" to see a list of the available reports that can be printed. This list will vary, depending on the project type, as follows:

      Acquisition Status Journal
        Acquisition Summary
         
      Disposal Status Journal
        Disposal Summary
        Fact Sheet
        First Nations Consultations
         
      Consulting Status Journal

    To print information about many projects, the user must create the list of projects they want to include in the report by using the Project Search and selecting the appropriate criteria (see above). Then the user must click on "Print" and click on the report of their choice. The three reports that are available are "SCH Report", "General Report" and "Project Summary". By clicking on "SCH Report" or "General Report", the user is taken to another page (SCHReportCreate.cfm or generalReportCreate.cfm) where they can choose the sort and enter other data that will appear on the report. The user must then click on "Submit" to create the report which can be printed from the browser. If the user chooses the "Project Summary" report (projectSummaryReport.cfm), the report is immediately created and can be printed from the browser. For this report the projects are sorted in the same order as the Search Results.

  • Admin

    The Admin section is where users can add, remove and edit items in the drop-down lists that appear in the project pages. To access the Admin section, users must be logged in and then click on "Admin" at the top of every page. Users with both read/write and administrative access can edit the "Location", "Client/Department" and "Client Contact" lists, as well as change their passwords and download data from the database in ASCII text format. In addition to this, users with administrative access can also edit the "Advisors", "Property Types", "Interest Sold Types", "Disposal Methods", and "Purchaser Types" lists. Admin users can also delete projects from the system, provided the project does not have any First Nations Consultation data.

Notation for the "QueryString Parameters and Form Fields" lists below:
x, y - both x and y are included
x | y - either x or y is included
[x] - parameter(s) is optional
x("X") - parameter x has value X

Example:
URL.projectID, URL.view, [Form.appraisalAdd | Form.appraisalUpdate]

Reads:
URL.projectID and URL.view are both passed and either Form.appraisalAdd or Form.appraisalUpdate may be passed.

Note: Only parameters and fields on which critical decisions are based are listed.


acquisitions.cfm

acquisitions.cfm displays the project information for acquisition projects. When the user clicks on the "Details" button of an acquisition project from the Search Results, acquisition.cfm is called. It reads the Project ID from the querystring to determine which project should be displayed and then retrieves the data from tblAcquisitions and tblProjects and fills in the fields with this data. acquisitions.cfm can be requested from many other files and depending on the parameters passed to it, it may perform various update and append queries. When acquisitions.cfm is requested by projectList.cfm or newProject.cfm it saves all of it's data in Session variables so that if the user clicks "Cancel" projectList.cfm can restore all the data to it's original state.

addOptions.cfm

When a user is adding a new project they have the option of adding new locations, client/departments and client contacts. When they click on a button to add one of these options, the form on newProject.cfm is sent to addOptions.cfm, where the values are entered into hidden form fields. Then when the user has filled out the form on addOptions.cfm, it (along with the hidden form fields) is sent back to newProject.cfm so that no data is lost.

When the user clicks on one of the "Add Option" buttons on newProject.cfm, the option type is passed to addOptions.cfm in Form.optionType. addOptions.cfm displays the appropriate form depending on the value in Form.optionType.

Security Setup and User Access

There are three levels of user access:

  • Read (read only)
  • Write (read/write)
  • Admin (read/write/admin)

Read: Users do not need an account to gain Read access, but there is a Read classification for accounts of advisors who no longer work with REAS. The accounts still exist because these advisors' names are still associated with projects, so their record must remain in tblAgent. Users with Read access can look at all data and print all reports but they cannot edit or add any data.

Write: To gain Write access, users must have a Write access account and be logged in to the system. Once a user is logged in they will remain so even after leaving and returning to the system, provided the user is accessing the system with the same computer and web browser and they have not manually logged out. With Write access users can edit data for all projects as well as add new projects. Write access users can also use some of the features in the Admin section; they can add, edit and delete departments, locations and client contacts, as well as change their passwords and download data from the database in ASCII comma delimited text format.

Admin: Admin access works the exact same as Write, except that Admin users can access all of the features in the Admin section. In addition to the Admin features mentioned for Write access, Admin users can also add, edit and delete advisor accounts, disposal methods, interest types, property types and purchaser types, as well as delete projects.

Maintenance Procedures

Performing these procedures will require a decent understanding of HTML, ColdFusion, JavaScript and possibly Macromedia Fireworks. This section is only meant to be used as a guide and does not provide the details of each step.

Add a field to a project type
In the description below we are adding a field called "xxxxx" to the Acquisition project type. Substitute "Disposal" or "Consulting" in place of "Acquisition" if you are adding a field to one of these project types.

  1. Add the field to the database (either tblAcquisitions or, if other project types will use this field as well, tblProjects).**
  2. In acquisitions.cfm:
    • Add the field name to the SELECT statement of the query, "qryDisplayProject".
    • Add two statements to the JavaScript function, "sendForm". One for Netscape browsers and one for Internet Explorer browsers.
    • Add two <CFSET> statements between the <CFOUTPUT QUERY="qryDisplayProject"> tags. One to set a variable called "projxxxxx" and one to set a variable called "session.backupxxxxx".
    • Add a hidden field to the form, "mainForm". If the field is required or must be a certain datatype add extra hidden fields to validate this.
    • Add a new field to the form on the layer (tab) of your choice. Set the value of the field equal to "projxxxxx".
  3. In acquisitionSave.cfm:
    • If xxxxx is a currency field, add statements to format it properly.
    • Add a line to the SET statement of the query, "qryUpdateProject", if the field is in tblProjects or the query, "qryUpdateAcquisition", if the field is in tblAcquisitions.
  4. In projectList.cfm:
    • Add a <CFSET> statment after the statement, <CFELSEIF IsDefined("URL.cancelType") AND URL.cancelType EQ "acquisition">, that sets the variable, "tempxxxxx", equal to "Session.backupxxxxx".
    • Add a line to the SET statement of the query, "qryUpdateProject", if the field is in tblProjects or the query, "qryUpdateAcquisition", if the field is in tblAcquisitions.

** There are two extra fields already in the database if they need to be used. They are strExtraField1 and strExtraField2. Both fields are in the tblProjects table and are text fields with a length of 255 characters.


Add a tab to a project type:
In the description below we are adding a tab called xxxxx which will be the nth tab on the acquisition projects page. Substitute "Disposal" or "Consulting" in place of "Acquisition" if you are adding a tab to one of these project types.

  1. Using Macromedia Fireworks open tabs.png:
    • Create two tab images, one blue and one grey, using the other tabs as templates.
    • Export (using a slice) the blue tab as "images/tab_xxxxx_on.gif"
    • Export (using a slice) the grey tab as "images/tab_xxxxx.gif"
  2. In acquisitions.cfm:
    • Add a style definition called #tabn with the visibility set the same way as the other tabs except the <CFIF> statement will read '... AND #URL.view# EQ "xxxxx">'.
    • In the JavaScript section of the page:
      • define a tabButton object the same way as the others:
        var tabButtonn = new tabButton("tabButton_n", "images/tab_xxxxx_on.gif"...
      • add a <CFELSEIF> statement for the new tab where "lastTabButton" is being set:
        <CFELSEIF #URL.view# EQ "xxxxx">tabButtonn</CFIF>
      • in function tabChange( ) add a statement to set "daTabButton" to tabButtonn
      • in function tabToggle( ) add a statement:
        document.all('tabn').style.visibility = hidden;
      • add statements to function sendForm( ) if required.
    • Add a column to the second table on the page with a hyperlinked image in the first row and a transparent image in the second row (similar to the other tabs).
    • Define a new layer on the page and fill it with content between the two <DIV> tags:
      • <DIV ID="tabn" class="tab">      </DIV>
